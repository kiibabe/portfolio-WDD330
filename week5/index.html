Chapter 10: Testing and Debugging

  • Types of error
    • System error: There is problem with the system or external devices with which the program is interacting
    • Programmer error: The program contains incorrect syntax or faulty logic (It could even be as simple as a typo)
    • User error: The user has entered data incorrectly, which the program is unable to handle
  • Exception: An error that produces a return value that can then be used by the program to deal with the error
  • Stack Trace, which an exception will also produce can help us debug. For example, it can help us find where an error originates from.
  • We need to ensure that the code we write fails loudly because failing silently makes errors difficult to spot and longer to track down.
  • Strict mode encourages a better quality of JavaScript that befits a programmer, so its use is recommended. It requires the following code to be added at the beginning of a javascript file: 
         'use strict';
  • Debugging in browser:
    • alert(): alert() stops a program from running until OK is clicked. In the past, using alerts for debugging was the only option 
           function isThisReality(reality){
        if (reality = false) {
            alert(reality);
            return 'No, you're in a simulation.';
        }
        else {
            return 'Yes, this is the reality.';
            }
        }
    • Using console:
      • console.log(): This is like alert() but it does not stop the program
      • console.trace(): This will show the functions that were called in the lead up to an exception occurring while the code is running
    • "debugger" command: This will create a breakpoint that will stop the program and allow us to see where the program is currently at. We shouldn't forget removing any debugger keyword before shipping, or it will freeze the program when people try to use it 
           function isThisReality(reality){
        debugger;
            if (reality = false) {
                return 'No, you're in a simulation.';
            }
            else {
                return 'Yes, this is the reality.';
            }
    }
    • Error objects:
      • RangeError: A number is outside an allowable range of values
      • ReferenceError: A reference is made to an item that doesn’t exist
      • SyntaxError: Error in the code syntax
      • TypeError: Error in the type of value used
      • URIError: There’s a problem encoding or decoding the URI
      • InternalError: A non-standard error. an error occurs in the JavaScript engine, like too much recursion
  • "Linting tools" like JS Lint, JS Hint, andES Lint can be used to test the quality of code
  • console.trace(): Logs an interactive stack trace in the console. This will show the functions that were called in the lead up to an exception occurring while the code is running
  • Exceptions can be thrown using the "throw" statement
  • Any code placed inside a "try" block will pass any error objects to a "catch" block when an exception occurs. Any code inside a "finally" block will run if an exception does or does not occur
  • Test-driven development: The practice of writing tests that fail, then writing the code that passes the test, then refactoring the code every time a new feature is implemented
