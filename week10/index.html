Validating Forms

  • Client side validation can be done via HTML5 validation features or JS
  • required, min/max, type, and pattern are examples of build in form validation
  • required makes the input mandatory, so the form cannot be submitted until filled
  • pattern can match input with regular expressions
  • min/max length will restrain the number of characters that can be entered into a text field
  • The constraint validation API in JS can be used to control the feel of error messages or be compatible with old browsers
  • ValidationMessage: sends message about which validations failed
  • Validity: returns if the input is valid based on different criteria
  • Some of the validation include patternMismatch (pattern), tooLong/tooShort (min/maxlength), typeMismatch (type), and valueMissing (required)
  • willValidate: returns true if value will be validated
  • checkValidity(): checks if element is valid, and returns a boolean value
  • reportValidity(): uses events to report elements that are invalid
  • setCustomValidity(): allows user to set custom error messages for failed validations that normally aren’t part of the HTML5 standard form validations
  • Error messages can be customized using CSS and :invalid pseudo-class
  • If the constraint validation API doesn’t fit your use case (like adding custom controls), then stand alone validation can be implemented in JS directly

Using Fetch

  • Fetch API helps create and receive HTTP requests and responses respectively
  • Basic fetch structure: 
      fetch('http://example.com/api).then(response => response.json()).then(data => console.log(data));
  • Fetch can also accept an init parameter that can be used to configure things like method, headers, etc.
  • Credentials can be set in init for cross origin and same origin calls
  • The body parameter can be included to send data in a POST or PUT request
  • If the input type is “file”, then that file can be sent as part of the body in the request
  • Check Response.ok to see if fetch was successful -- will return an error in case of CORS or network issues
  • Client and server side errors can be caught after returning the response object in case of a successful fetch
  • The Headers() constructor can be used to add and manage headers by appending and deleting them
  • The response object includes status code that tells us how the request was handled by the server
  • The body can accept a variety of types, like string, Blob (file), ArrayBuffer, FormData, and others
